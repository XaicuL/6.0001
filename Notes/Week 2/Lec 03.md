# **STRINGS: 데이터의 서열과 주소**

### **(1) Indexing (인덱싱): 위치의 특정**

- **메커니즘**: 문자열을 대소문자를 구분하는 캐릭터들의 **시퀀스(Sequence)** 로 취급한다.
    
- **[Check Point]**: 인덱싱은 언제나 **0**부터 시작하며 , 마지막 요소는 항상 **-1**이라는 주소를 가진다.
    
- **[Necessity]**: 전체 데이터가 아닌, 특정 지점의 객체에 접근하여 비교(`==`, `>`, `<`)하거나 추출하기 위함이다.
    
### **(2) Slicing (슬라이싱): 구간의 추출**

- **도구**: `s[start : stop : step]`
    
- **[Analysis]**:
    
    - 두 개의 숫자만 주어질 경우 `step`은 `1`로 고정된다.
        
    - `s[::-1]`은 전체 시퀀스를 역순으로 순회하며 새로운 문자열을 생성한다.
        
    - 범위를 벗어난 인덱싱은 에러를 유발하지만, 슬라이싱은 유연하게 작동한다.
    
예1) String Slicing: 인덱스의 변주와 객체 생성
```
>s = "abcdefgh" # index: 0 1 2 3 4 5 6 7 [cite: 64]

# Case 1: 기본 슬라이싱
>print(s[3:6])   # 'def' 

# Case 2: Step을 활용한 도약
>print(s[3:6:2]) # 'df' 

# Case 3: 음수 Step을 이용한 역방향 추출
>print(s[4:1:-2]) # 'ec' 
```
        
### **[Analysis]**: 슬라이싱 메커니즘의 엄밀성

- **[1] 구간의 정의**: `s[start:stop:step]` 구조에서 `stop` 인덱스는 포함되지 않으며, 
  `stop-1` 지점에서 추출이 종료된다.
    
- **[2] 음수 Step의 방향성**: `step`이 음수일 경우 시작점은 끝점보다 커야 하며, 
  인덱스는 왼쪽(역방향)으로 이동한다.
    
- **[Necessity]**: 파이썬의 문자열은 수정 불가능(**Immutable**)하기 때문에, 
  특정 부분을 바꾸고 싶다면 슬라이싱으로 필요한 부위를 도려내어 새롭게 바인딩하는 과정이 필연적이다


### **(3) Immutability (불변성)**

- **[Critical Point]**: 파이썬의 문자열은 **'Immutable'** 하여 직접적인 수정이 불가능하다.
    
- **해결책**: 기존 객체를 수정하는 것이 아니라, 
  슬라이싱과 결합을 통해 **새로운 객체를 생성하고 변수에 재바인딩**하는 방식을 취한다.

## **# 2. ALGORITHMS: 해결의 필연성**

### **(1) Guess-and-Check (Exhaustive Enumeration)**

- **정의**: 가능한 모든 값을 대입하여 정답을 확인하는 과정이다.
    
- **[Necessity]**: 해의 후보군이 명확할 때, 논리적 비약 없이 모든 가능성을 검토하여 해를 확정 짓기 위해 사용한다.

예1) Guess-and-Check: 전수 조사의 당위성
```
>cube = 8  
>for guess in range(abs(cube) + 1):  
>     if guess**3 >= abs(cube):  
        break  
>if guess**3 != abs(cube):  
>     print(f'{cube}는 완전 세제곱수가 아닙니다.')  
>else:  
>    if cube < 0:  
>       guess = -guess  
>    print(f'{cube}의 세제곱근은 {guess}입니다.')

8의 세제곱근은 2입니다.
```

### ### **[Analysis]**: 완전 탐색의 논리적 필연성

- **[1] 탐색 범위의 제한**: `range(abs(cube) + 1)`을 통해 해가 존재할 수 있는 물리적 한계를 설정한다.
    
- **[2] 단호한 중단 (`break`)**: `guess**3`이 목표치에 도달하거나 넘어서는 순간, 더 이상의 연산은 불필요하므로 즉시 루프를 탈출한다.
    
- **[Necessity]**: 정답이 정수임이 보장되는 상황에서는 복잡한 수학적 근사치 계산보다 모든 경우를 대입하는 것이 가장 확실하고 오류가 없는 해결책이다.
### **(2) Approximate Solutions (근사해)**

- **정의**: 완벽한 정답 대신, 허용 가능한 오차 범위(**Epsilon**) 내의 해를 찾는다.
    
- **[Analysis]**:
    
    - **Increment(증분)** 가 작을수록 정확도는 높아지나 프로그램의 속도는 저하된다.
    
    - **Epsilon**이 커질수록 속도는 빨라지나 결과의 정밀도는 낮아진다.
        
### **(3) Bisection Search (이분 탐색)**

- **메커니즘**: 매 반복마다 탐색 범위를 **절반**으로 축소하여 해에 접근한다.
    
- **[Necessity]**
  탐색 공간($N$)이 방대할 때, 기하급수적으로 연산 횟수를 줄여 효율성을 극대화하기 위함이다. 
  (수렴 속도: $\log_2 N$) .


예2) Bisection Search: 효율의 극치
```
>cube = 27
>epsilon = 0.01      # 허용 오차 
>low = 0             # 탐색 하한
>high = cube         # 탐색 상한
>guess = (high + low) / 2.0 # 중간값 선택 

>while abs(guess**3 - cube) >= epsilon:
>   if guess**3 < cube:
>       low = guess     # 하한선을 중간으로 이동
>   else:
>        high = guess    # 상한선을 중간으로 이동
>    guess = (high + low) / 2.0 # 새로운 중간값 갱신 

>print(f'{guess}는 {cube}의 세제곱근에 근접한 값입니다.') 

3.000091552734375는 27의 세제곱근에 근접한 값입니다.
```
### **[Analysis] : 이분 탐색의 수렴 논리**

- **[1] 오차 범위 ($\epsilon$)**: 완벽한 정답 대신 '충분히 좋은(Good enough)' 근사해를 구하기 위한 기준을 설정한다.
    
- **[2] 탐색 공간의 축소**: 매 회차마다 검색 영역을 절반씩 제거함으로써, $k$번의 시도 만에 $N/2^k$만큼의 정확도를 확보한다.
    
- **[Necessity]**: 데이터의 양($N$)이 방대할 때 순차 탐색은 성능적 한계에 부딪히지만, 이분 탐색은 $\log_2$ 단위의 수렴 속도를 통해 압도적인 효율을 보장한다


**[Remark] 알고리즘의 선택 기준**

- 정수와 같이 딱 떨어지는 해를 찾을 때는 **Guess-and-Check**가 유효하다.
    
- 연속적인 실수 범위에서 '충분히 좋은' 해가 필요할 때는 **Approximate Solution**을 선택한다.
    
- 함수의 값이 단조 증가/감소하는 특성을 가질 때, 압도적인 속도를 위해 **Bisection Search**를 꺼내 든다.
